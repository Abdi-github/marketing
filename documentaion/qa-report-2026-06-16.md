## Bug Report

### BUG-001 — HIGH (FIXED) — Form AI generator: `type: "date"` rejected by schema

| Field    | Value                                |
| -------- | ------------------------------------ |
| Module   | Forms                                |
| Route    | `/en/forms/new`                      |
| Role     | Any authenticated user               |
| Severity | HIGH — blocks entire AI form builder |
| Status   | **FIXED**                            |

**Steps to reproduce:**

1. Navigate to `/en/forms/new`
2. Enter a description mentioning a "date" field (e.g., reservation form with preferred date)
3. Click "Generate with AI"

**Expected:** Form schema generated successfully

**Actual:** 500 error — "AI returned an invalid form schema"

**Root cause:** The `form-builder-v1` prompt listed general rules but didn't enumerate the allowed field types. Claude Haiku correctly chose `type: "date"` for a date input, but the `FORM_FIELD_TYPES` enum (`text|email|tel|textarea|select|radio|checkbox|number`) doesn't include `date`. Zod validation rejected the response.

**Fix:** Updated `packages/ai-router/src/prompts/registry.ts` — the `form-builder-v1` system prompt now explicitly lists all 8 allowed types with a note: _"there is no 'date' type — use type 'text' with placeholder 'DD.MM.YYYY' for date fields."_

**File changed:** [packages/ai-router/src/prompts/registry.ts](../packages/ai-router/src/prompts/registry.ts)

### BUG-002 — HIGH (FIXED) — Embed form 404 due to i18n middleware locale prefix

| Field    | Value                                       |
| -------- | ------------------------------------------- |
| Module   | Forms (public embed)                        |
| Route    | `/embed/forms/[tenantSlug]/[formSlug]`      |
| Role     | Public (no auth)                            |
| Severity | HIGH — makes all form embeds non-functional |
| Status   | **FIXED**                                   |

**Steps to reproduce:**

1. Copy embed code from a form's detail page
2. Navigate to `http://[host]/embed/forms/[tenantSlug]/[formSlug]`

**Expected:** Form renders in embed-friendly layout

**Actual:** Redirected to `/en/embed/forms/...` → 404 (no locale-prefixed embed route exists)

**Root cause:** The Next.js i18n middleware (`next-intl`) was applying locale prefix to all platform-host requests without exception. The `/embed/` app directory routes (`apps/web/src/app/embed/`) are not locale-prefixed in the filesystem, so they 404 when accessed with `/en/` prefix.

**Fix:** Added an early bypass in `apps/web/src/middleware.ts`: requests to `pathname.startsWith('/embed/')` call `NextResponse.next()` directly, skipping the `intlMiddleware`.

**File changed:** [apps/web/src/middleware.ts](../apps/web/src/middleware.ts)

### BUG-006 — INFO — Sandboxed iframe console errors on template gallery

| Field    | Value                   |
| -------- | ----------------------- |
| Module   | Landing pages           |
| Route    | `/en/landing-pages/new` |
| Severity | INFO — not a real bug   |

**Observation:** 42–114 "Blocked script execution in sandboxed iframe" errors appear in the browser console when viewing the template gallery.

**Cause:** Template previews are rendered in correctly sandboxed `<iframe>` elements (security feature). Scripts inside the previews are intentionally blocked. Visual content still renders correctly.

**No fix needed.** This is correct security behavior. The noise could be reduced by adding an appropriate `allow-scripts` sandbox attribute only if the preview iframes require script execution, but current behavior is safe.
